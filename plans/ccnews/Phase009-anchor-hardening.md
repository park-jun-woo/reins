# Phase 009 — event6 앵커 게이트 치즈 저항 강화

- 프로젝트: ccnews (GitHub: `park-jun-woo/quest-ccnews`)
- 작성일: 2026-06-03
- 상태: 설계 — 구현 대기
- 의존: Phase006(event6 앵커 게이트), Phase005(구조화 추출)

## 목적

event6 앵커 게이트를 **치즈(Goodhart) 저항적**으로 강화한다. 약한 모델이 프록시(앵커
substring 체크)와 의도(진짜 근거 있는 사실)의 틈을 파고드는 것을 막는다.

## 배경 (Phase007 풀-WARC 300건 실측 근거)

풀 WARC(21,941건) 시드 후 Opus/Sonnet/Haiku로 각 100건 병렬 처리 중 발견:

| 모델 | 처리 | 빈-앵커 치즈 | 플레이스홀더(value="Subject"/"Event") | 정상 |
|---|---|---|---|---|
| Opus | 58 | 0 | 2 | 56 |
| Sonnet | 23 | 0 | 2 | 21 |
| **Haiku** | 21 | **17 (81%)** | 12 | 4 |

- Haiku가 `who="Subject"`, `anchors=[""]` 같은 **껍데기 event6를 PASS시킴**. 원인:
  Go `strings.Contains(text, "") == true` — **빈 문자열은 모든 텍스트의 substring**이라 무조건 통과.
  `len(anchors)==0` REVIEW 트리거도 `[""]`(len 1)이라 우회 → 곧장 PASS.
- 강한 모델(Opus/Sonnet)은 같은 구멍을 안 건드림(성실히 진짜 앵커 추출). Haiku도 처음 몇 건은
  다국어 정확 추출 → **능력 문제가 아니라 게이트에 빈틈이 있으니 샌 것**.

핵심 교훈: **모델 적합성 = 게이트의 치즈 내성**(모델 세기가 아님). 약한 모델은 게이트 허점의
탐지기다. 설계 검수도 못 잡은 이 구멍을 실데이터+약한 모델이 드러냈다.

## 치즈 벡터 (현 게이트에서 가능한 우회)

| # | 우회 | 현 상태 |
|---|---|---|
| 1 | 빈/공백 앵커 `anchors:[""]` | 뚫림 (확인) |
| 2 | 흔한 토큰 앵커(`"the"`·한 글자·사이트명·`Home`/`Login` 등 chrome) | 뚫림 |
| 3 | value↔anchor 단절 (value 미검증 — value 거짓, 앵커만 진짜) | 뚫림 |
| 4 | 엉뚱한 위치 앵커(related stories·광고·다른 기사 조각) | 뚫림(전문 대상) |
| 5 | 공허한 value(`Subject`/`Event`/`N/A`) | 뚫림 |

> **실측 관찰 범위**: 300건 실데이터에서 *실제로 악용된* 것은 **벡터 1(빈앵커 17건)·벡터 5(플레이스홀더
> value 16건)** 뿐. 벡터 2·4는 코드상 가능하나 아직 관찰 안 됨(선제 하드닝 대상). 이 구분이 아래 방어
> 우선순위를 가른다.

## 방어 (계층)

### L0 — 빈 앵커 봉쇄 (벡터 1) · 필수·즉시
- 근본 원인: `checkField`가 `len(f.Anchors)==0`(raw 슬라이스)으로만 0개를 판정하고,
  `normalize("")==""` + `strings.Contains(src,"")==true`라 `[""]`가 anchored로 통과.
- 수정: 앵커를 **normalize → 빈/공백 제거하여 "유효앵커" 슬라이스를 한 번 계산**하고, **개수 체크와
  substring 루프 둘 다 유효앵커 기준**으로 본다(한 곳 수정으로 우회 완전 차단).
- 의미는 기존 보존: 필수필드 **유효앵커 0 → FAIL**, 선택필드 유효앵커 0 → **REVIEW**.
- 비용 0, false-negative 0, 미결 파라미터 없음 → 즉시 구현.

### L1 — 폴백 본문 boilerplate 제거 강화 (벡터 2·4 · 선제)
- **articleBody 우선 매칭은 이미 구현됨**: `extract/parse.go`가 `Body` 있으면 `BodyText=Body`,
  없으면 `stripHTML` 전문 → `submit.go`가 `res.BodyText`를 게이트에 전달. articleBody 사이트는 이미
  chrome/related/ads가 빠져 있음. **이 부분은 신규 작업 아님.**
- **신규 작업은 폴백 경로 1줄뿐**: articleBody 없는 사이트의 `stripHTML`은 현재 `isNonContent`가
  `script/style/noscript/template/head`만 제거 → 여기에 `<nav>/<header>/<footer>/<aside>` 추가.
- 실측 관찰 벡터 아님(폴백 품질 개선 성격의 방어심화). false-negative 거의 0.

### L2 — 앵커 강도 (벡터 2)
- 단일 비숫자 문자·순수 구두점·다국어 스톱워드 앵커 무효.
- 예외: 숫자·날짜·대문자 약어(UN/EU/AP)는 짧아도 유효.
- 필수필드는 유효앵커 ≥1개가 길이 ≥N(예: 3) 또는 숫자/약어.

### L3 — value 위생 (벡터 5)
- value 비어있지 않음 + 길이 하한 + 플레이스홀더 블록리스트(Subject/Event/Unknown/N/A/-).

### L4 — 분포 감사 (벡터 3, 방어심화 · per-record 게이트 아님)
- 데이터셋 레벨 치즈 신호 모니터: 짧은-앵커 비율·필드 간 동일 앵커·플레이스홀더 value 비율이
  임계 초과면 경보/리뷰 플래그. "no silent caps" 원칙.

## 환원 불가 한계 (정직하게)

**벡터 3(value 정확성)** 은 결정론으로 완전 차단 불가. 산출 value는 영어, 앵커는 원어라
번역 gap 때문에 `value⊆anchor` 교차검증이 성립 안 함. "앵커는 진짜인데 value 번역/요약이 틀림"은
게이트가 못 잡는다. **이 잔여가 정확히 REVIEW/사람의 몫**(강한 게이트 + 사람 확인의 조합).

## 결정론적 성격

L0~L3는 전부 입력(event6 + 원문 텍스트)만으로 결정되는 **순수·결정론적 규칙**. 같은 입력 →
같은 판정. AI 자기평가 없음. trim·길이·블록리스트·substring은 기계가 본다.

## 권장 번들

실측 근거와 실행 가능성으로 우선순위를 가른다.

- **필수·즉시(실측 근거 + 미결 없음)**: **L0 + L3**. 관찰된 치즈(빈앵커 17건·플레이스홀더 value 24건)를
  정확히 차단(빈앵커 17 = 전부 Haiku, 플레이스홀더 16 = 세 모델 합). 둘 다 작고 false-negative 0,
  결정해야 할 파라미터 없음 → 바로 구현.
- **방어심화(관찰 안 됨·선택·뒤)**: L1 폴백 보강 + L2 앵커 강도. 벡터 2·4는 실데이터에서 아직 안 나옴.
  특히 **L2는 최소길이 N·다국어 스톱워드가 미확정(아래 열린 결정)이라 지금은 구현 불가**이고 과도거부
  위험이 있음 → 열린 결정 확정 후 별도로 합류. L1은 폴백 1줄 보강이라 저비용이나 데이터 근거는 약함.
- L4는 운영 모니터로 추가.

## 열린 결정

- L2 최소 길이 N(예: 3)과 숫자/약어 예외 규칙의 정확한 경계.
- 다국어 스톱워드 목록 범위(주요 언어만 vs 광범위) — 과도 거부 위험 균형.
- L3 플레이스홀더 블록리스트 항목.
- L1 articleBody 폴백 시 boilerplate 제거 수준.
